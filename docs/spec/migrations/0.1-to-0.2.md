# Spec migration: v0.1 → v0.2

> **Type**: minor bump (additive only)
> **Breaking**: none for compliant clients
> **Effort**: low (新 field を ignore できる client なら無作業、 新機能 opt-in 制)

v0.2 は v0.1 の self-review (gap G1-G14) で見つかった **未記述 (not 誤り)** を埋める additive enrichment。 archive: [`archive/world-tree-v0.1.kdl`](../archive/world-tree-v0.1.kdl)

## TL;DR

- v0.1 client / impl は **そのまま動作する** (新 block / field を無視するだけ)
- v0.2 機能を使いたい場合は新 block / endpoint / scope を opt-in
- enum value 追加が複数あるが **open enum convention** (G10) に準拠していれば問題なし
- dangling ref (`actor` → `vp-actor`) 修正は v0.1 で実装してた consumer がいないので無影響

## 変更カテゴリ別 migration ガイド

### 1. New blocks (purely additive)

以下の top-level block が v0.2 で新規追加。 既存 client は ignore で OK:

- `spec-syntax`
- `spec-lint`
- `spec-versioning`
- `identity-delegation`
- `handle-namespace`
- `path-schema` (v0.1 の同名 block を expansion)
- `api`
- `api-response-conventions`
- `tombstone-policy`
- `gdpr`
- `product-manifest`
- `observability`
- `discovery`
- `phase-scoping`

### 2. Sync protocol expansion (G3)

v0.1:
```kdl
sync-protocol {
    default "event-sourced"
    fallback "webhook"
    consistency "eventual"
    event-schema { ... }
}
```

v0.2 (新規 child block):
- `delivery-guarantee "at-least-once"`
- `dedup-strategy` / `dedup-scope` / `dedup-ttl-hours`
- `ordering` block
- `signing` block
- `retry` block
- `replay` block
- `backpressure` block

**Action for existing impl**:
- product 側: `sequence_number` / `resource_id` / `signature` / `signature_ts` を event publish 時に同梱推奨 (Phase 1 任意、 Phase 2 必須)
- consumer 側: `delivery-guarantee=at-least-once` 前提で idempotent handler を確認

### 3. Event schema 拡張 (G3, G4)

新 field (全て optional / additive):
- `sequence_number` int — Phase 2 で required 化予定、 Phase 1 任意
- `resource_id` string — 同上
- `prev_event_id` uuid — 任意
- `signature` / `signature_ts` — Phase 2 で ingress mandatory、 Phase 1 任意
- `deleted_at` / `deleted_reason` / `final_payload` — `resource.deleted` 時のみ
- `replay` bool default=#false

**Action**: product 側で event publisher を更新、 missing field は default で受理される。

### 4. Resource-base 拡張 (G4, G8)

```kdl
// v0.1 → v0.2 diff
resource-base {
    field "owner" ref target="user" key-field="usr_id"   # ← key-field 明示 (G8)
    field "deleted_at" timestamp                          # ← 新規 (G4)
}
```

**Action**: DB schema migration で `deleted_at` column 追加。 既存 row は NULL (alive)。

### 5. Auth block 大幅 expansion (G9)

v0.1:
```kdl
auth {
    oidc-issuer ...
    product-token { ... }
    user-token { ... }
}
```

v0.2:
- `audiences` を **list 化** (string → array)、 単一値はそのまま array に migration
- `scope-registry` を **dotted notation** で全置換 (G5)
- `user-token.refresh-flow` / `revocation-flow` / `revocation-cache-ttl-seconds` 追加
- `product-token.ttl-days` / `rotation-reminder-days` / `rotation-overlap-days` 追加
- `app-user-token` 新規 (Phase 2)
- `introspection-endpoint` / `revocation-list-fetch-interval-seconds` / `jwks-cache-ttl-seconds` 追加
- `least-privilege-rule` / `scope-discovery-endpoint` 追加

**Action**: 既存 token は v0.1 形式で発行されたまま valid。 新 token から audience 配列で発行可能。 scope は v0.2 形式に gradual migration。

### 6. Scope refinement (G5)

v0.1 product manifest:
```kdl
product "creo-memories" {
    scope "register:resource"
    scope "delete:resource"
}
```

v0.2:
```kdl
product "creo-memories" {
    scope "events.publish.memories-atlas"
    scope "events.delete.memories-atlas"
}
```

**Action**:
- 既存 product-token は v0.1 scope のまま **互換 alias** で受理 (`register:resource` ⇔ `events.publish.*`)
- 新 token issuance / rotation 時に dotted notation に switch
- `cplp` の delete scope 復活なし、 `fleetstage` / `gfp` の delete scope は新規発行

### 7. Resource-type changes

#### vp-world (G7)

v0.1:
```kdl
field "actors" list { element ref target="actor" }   # ← actor 未定義 (dangling)
```

v0.2:
```kdl
field "actors" list { element ref target="vp-actor" }
```

**Action**: 既存 producer は `target="actor"` の event を emit してなかった (dangling だったため)。 v0.2 で `vp-actor` resource を別途 publish。

#### identity subtree (G1, G7)

v0.1: `subtree { child "identity" cardinality="1" owner="creo-id" }` — resource-type 不在

v0.2: virtual subtree、 `external-source "creo-id"` + redirect URL

**Action**: 既存 `GET /users/{handle}/identity` は **307 Temporary Redirect** で Creo ID へ。 client は redirect-follow が必要。

#### user.payload.account_type (G12)

新規 field、 default `"user"`。 既存 user record は default 値で初期化 (DB migration)。

#### 新 resource-types

- `vp-actor` (G7、 Phase 1)
- `organization` (G12、 Phase 2 impl)
- `org-membership` (G12、 Phase 2 impl)
- `notification` (G14、 Phase 3 impl)
- `follow` (G14、 Phase 3 impl)

Phase 2/3 のものは spec 宣言のみ、 impl 未着手で OK。

### 8. Path schema 拡張 (G11)

reserved-path-slug が拡大: `api`, `oauth`, `errors`, `static`, `health`, `_`

**Action**: 既存 user の handle にこれらが衝突していないか確認 (なし、 reserved-handles で予約済)。

### 9. Codegen spec-version pinning (G10)

emit artifact に spec version embed:
- TS: `export const SPEC_VERSION = "0.2.0"`
- Rust: `pub const SPEC_VERSION: &str = "0.2.0"`
- Ruby: `ChronistaHub::SPEC_VERSION = "0.2.0"`

**Action**: codegen を re-run、 SDK を 0.2.0 として release。

## DB migration scripts (例)

```surql
-- migrations/002_spec_0_2_0.surql

-- G4: tombstone field
DEFINE FIELD OVERWRITE deleted_at ON resource TYPE option<datetime>
    PERMISSIONS FULL;

-- G12: account_type
DEFINE FIELD OVERWRITE account_type ON user TYPE string DEFAULT "user"
    ASSERT $value IN ["user", "organization", "reserved"]
    PERMISSIONS FULL;

UPDATE user SET account_type = "user" WHERE account_type IS NONE;

-- G12: reserved handle sentinel records (release 前に投入)
CREATE user CONTENT {
    handle: "chronista-club",
    account_type: "reserved",
    email: "system@chronista.club",
    created_at: time::now(),
};
-- ... 他の reserved handle も同様
```

## Rollback

破壊的変更が無いので rollback は spec ファイル revert で完了。 DB の `deleted_at` / `account_type` column は残しても無害 (新 column は使わなければ無視される)。
